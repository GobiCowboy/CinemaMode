# 104 技术选型

## 1. 技术选型表

| 类别 | 选择 | 用途 | 不选其他方案的原因 | 注意事项 |
|------|------|------|--------------------|----------|
| 语言 / 运行时 | Swift | 原生 macOS 应用开发 | Electron、Tauri 会增加体积和系统控制复杂度 | 使用现代 Swift 并保持代码分层 |
| UI 框架 | SwiftUI | 主窗口、设置页、入口、浮窗内容 | 纯 AppKit 开发成本更高，普通 UI 不需要 | 系统级窗口行为通过窄 AppKit 桥接 |
| 平台桥接 | AppKit | presentation options、设置窗口、浮动 panel、鼠标事件 | SwiftUI 无法完整覆盖系统级显示控制 | 桥接层集中在 `Platform/` |
| 数据存储 | UserDefaults / AppStorage | 保存轻量偏好和上次状态标记 | SQLite、Core Data 对 MVP 过重 | 不保存网页、播放器或用户内容 |
| 日志方案 | OSLog + 项目 Logger 封装 | 结构化记录启动、进入、退出、恢复 | `print`/`console.log` 无模块和上下文 | 生产日志不记录敏感内容 |
| 构建工具 | Xcode 项目或 Swift Package + Xcode 工程 | 本地构建、签名、发布 | 自定义脚本不作为主要构建入口 | 后续实现阶段确定最终工程形态 |
| 测试工具 | XCTest | 服务层状态机、平台桥接可替身测试 | 手动测试不能覆盖异常恢复 | 平台层保留协议方便 mock |
| 发布形态 | `.app`，后续可 DMG | 面向普通 Mac 用户 | CLI 或脚本不符合产品定位 | 发布前需处理签名、公证 |

## 2. 关键取舍

| 编号 | 选择 | 放弃 | 原因 | 后续风险 |
|------|------|------|------|----------|
| T-001 | 原生 SwiftUI + AppKit | Electron / Web 技术 | 产品需要轻量、原生、低权限和系统显示控制。 | 需要熟悉 macOS 桥接和窗口生命周期。 |
| T-002 | 用户手动触发 | 自动识别视频 | 避免监听、误判和复杂权限说明，符合“用户主动决定”。 | 用户仍需要主动点击一次。 |
| T-003 | 极简浮窗退出 | 快捷键作为主退出方式 | 普通用户不应记快捷键，退出入口必须可见。 | 浮窗层级和透明度要精心验证。 |
| T-004 | 轻量偏好存储 | 数据库 | MVP 不需要复杂数据实体。 | 后续如果有场景规则需重新评估。 |
| T-005 | 窄 AppKit 桥接 | 全 AppKit 应用 | 保留 SwiftUI 开发效率，只在必要位置使用 AppKit。 | 桥接边界必须严格，避免平台代码扩散。 |
| T-006 | 独立设置页 | 直接塞进主入口 | 设置需要可发现、可回访、可持续保存。 | 需要单独维护窗口生命周期。 |

## 3. 锁定技术

| 技术 | 锁定原因 | 解禁条件 |
|------|----------|----------|
| Swift | 原生 macOS 能力和发布体验最佳。 | 证明原生方案无法满足核心体验。 |
| SwiftUI | MVP UI 简单，适合快速迭代。 | SwiftUI 严重阻碍窗口或浮窗体验。 |
| AppKit presentation options | 菜单栏和 Dock 控制必须使用系统级能力。 | Apple 提供更合适的新 API。 |
| NSPanel / NSWindow 浮动窗口能力 | 退出入口必须独立于主窗口并始终可见。 | SwiftUI 提供等价、稳定的浮窗 scene 能力。 |
| OSLog 封装 | macOS 原生日志能力，便于调试和发布。 | 引入更强日志系统且不增加用户负担。 |

## 4. MVP 技术边界

| 事项 | 决定 |
|------|------|
| 不请求辅助功能权限 | MVP 不做跨应用控制、按键注入或播放器监听。 |
| 不安装浏览器插件 | 保持安装和理解成本最低。 |
| 不读取网页内容 | 避免隐私和适配成本。 |
| 不改变播放器状态 | Cinema Mode 只管理系统沉浸环境。 |
| 不默认隐藏退出入口 | 透明度最低约 5%，但窗口仍存在。 |
| 不把偏好塞进主流程 | 设置页单独承载音量、语言等偏好。 |
